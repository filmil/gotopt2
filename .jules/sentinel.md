## 2025-01-31 - [Command Injection via Double Quotes in eval Payload]
**Vulnerability:** Command injection was possible because `gotopt2` output strings wrapped in double quotes (e.g., `gotopt2_var="$(echo hacked)"`), which caused the host bash script using `eval` to evaluate the inner command substitution.
**Learning:** When generating shell code intended for `eval`, double quotes are unsafe if the content includes user input, because they allow parameter expansion and command substitution.
**Prevention:** Always wrap generated string values in single quotes, and escape internal single quotes as `'"'"'` (or similar), to ensure the string is treated purely as a literal value by the shell parser.
## 2025-02-05 - [Command Injection via Configuration Parameters in eval Payload]
**Vulnerability:** Command injection was possible because `gotopt2` output bash variables without sanitizing the `name`, `prefix`, and `declaration` configuration inputs. An attacker could inject arbitrary commands by providing values like `foo;echo INJECTED`.
**Learning:** Even configurable prefixes and variable names can be vectors for command injection if the output is evaluated (`eval`). Unsanitized configuration strings can break out of variable assignment contexts.
**Prevention:** Strictly sanitize all variables, prefixes, and declarations using an allowlist approach to ensure only valid Bash identifier characters (`[a-zA-Z0-9_]`) or specific declaration characters (`[a-zA-Z0-9_ \-]`) are emitted into shell scripts.
## 2025-02-05 - [Command Injection via Configuration Parameters in gotopt2-generator]
**Vulnerability:** Command injection was possible because `gotopt2-generator` output bash scripts without sanitizing the `name` and `prefix` configuration inputs. An attacker could inject arbitrary commands directly into the generated script by providing values like `evil;echo INJECTED`.
**Learning:** Configurable prefixes and variable names can be vectors for command injection if the output is evaluated or sourced as a script. Unsanitized configuration strings can break out of variable assignment contexts in the generated file.
**Prevention:** Strictly sanitize all variables, prefixes, and declarations using an allowlist approach to ensure only valid Bash identifier characters (`[a-zA-Z0-9_]`) are emitted into shell scripts.
## 2025-02-05 - [Command Injection via Unquoted Default Values in gotopt2-generator]
**Vulnerability:** Command injection was possible in `gotopt2-generator` because the `DefaultValue`, `TrueValue`, and `Help` configuration fields were interpolated directly into the generated `bash` and `fish` shell scripts without adequate shell quoting. An attacker could embed commands in the YAML configuration (e.g., `default: '$(touch /tmp/pwned)'`), leading to arbitrary execution when the script is sourced or evaluated.
**Learning:** Hardcoding string interpolations directly into double-quoted shell echoes or variable assignments (e.g., `VAR="{{.Value}}"`) introduces injection risks, as parameter expansion and command substitution occur inside double quotes.
**Prevention:** Always use dedicated shell-escaping functions (such as Go's internal `shellQuote` wrapper for `bash` or `fishQuote` for `fish` to replace `\` and `'`) to securely wrap generated strings in single quotes before interpolation into templates.
## 2024-06-28 - Command Injection via Unescaped Template Variable

**Vulnerability:** The `.Usage` text from the YAML configuration was directly injected into double quotes (`echo "{{ .Usage }}"`) inside the generated `gotopt2-generator` shell scripts. Since double quotes evaluate shell substitution sequences like `$(...)` and ``` `...` ```, this allowed malicious `usage` text to execute arbitrary commands when users ran `--help` on the generated script.
**Learning:** Shell scripts generated from templates are highly susceptible to command injection if variables are improperly quoted. Escaping dynamic configuration strings accurately for different shell syntaxes (like fish vs bash) before injecting them into templates is crucial to prevent execution contexts.
**Prevention:** Always sanitize or safely quote inputs passed to script generators using library functions like `opts.ShellQuote` or custom implementations for non-standard shells before injecting them into templates, and avoid enclosing raw template variables inside double quotes in shell execution paths.
